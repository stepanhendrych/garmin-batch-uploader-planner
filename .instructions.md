## Project Overview

**Garmin Batch Uploader & Planner** is a Python CLI tool for bulk uploading and managing workouts in Garmin Connect. It supports JSON/CSV input, token-based authentication with MFA/2FA, and interactive management features like deleting workouts by sport type.

### Tech Stack
- **Language:** Python 3.10+
- **API Client:** `garminconnect` (with `garth` for HTTP requests)
- **CLI Framework:** `argparse` (no click/typer)
- **Data:**  JSON/CSV parsers, dataclass models
- **Authentication:** Token persistence via `garth`

### Key Files
- `cli.py` — Main entry point with argparse
- `garmin_batch_uploader/` — Package with modular components
  - `models.py` — `Workout` dataclass
  - `parsers.py` — JSON/CSV loaders
  - `garmin_client.py` — Garmin API wrapper
  - `planner.py` — Orchestration, interactive workflows
  - `config.py` — Environment settings loader

## Code Style & Conventions

### Python
- **Type hints:** Always use for function parameters and return types (`from __future__ import annotations`)
- **Logging:** Use Python's `logging` module (not `print`)
- **Docstrings:** Include for classes and public methods
- **Format strings:** F-strings preferred
- **Imports:** Organized per PEP 8 (stdlib, third-party, local)

**Example:**
```python
from __future__ import annotations

import logging
from pathlib import Path
from typing import Any

from garminconnect import Garmin

LOGGER = logging.getLogger(__name__)

class MyClass:
    """Brief description."""
    
    def method(self, param: str) -> bool:
        """What this does.
        
        Args:
            param: Description.
            
        Returns:
            Success status.
        """
        return True
```

### Design Patterns
- **Dataclasses:** Use `@dataclass(frozen=True)` for immutable models (see `models.py`/`Workout`)
- **Separation of Concerns:**
  - `models.py` — Data structures (no I/O)
  - `parsers.py` — File loading and validation
  - `garmin_client.py` — Garmin API interactions only
  - `planner.py` — Business logic and orchestration
  - `config.py` — Settings loading
- **Error Handling:** Raise `ValueError` for validation, log context before raising

### API Integration (Garmin Connect)
- Use `self._client.garth.request(method, subdomain, path)` for direct HTTP calls
- Example: `garth.request("DELETE", "connectapi", f"/workout-service/workout/{id}")`
- Always check status codes: `response.status_code in (200, 204)` for success
- Error responses are often 403 (Forbidden) — log and handle gracefully
- Token persistence: Tokens saved to `~/.garminconnect` automatically (don't hardcode paths)

## CLI & Interactive Features

### Command Structure
- Use `argparse.ArgumentParser` with `add_argument()` for flags
- Keep commands simple: `--file`, `--dry-run`, `--verbose`, `--delete-sport-type`
- All workflows should support `--dry-run` mode (mock API calls)

### Interactive Menus
For user prompts (like sport type selection):
- **Display options** with numbered list (1, 2, 3...)
- **Accept input** via `input("> ")` and strip/lowercase
- **Validate** and re-prompt on invalid input
- **Confirm destructive actions:** e.g., "Type 'yes' to confirm deletion"
- **Cancel option:** Accept 'q' or 'cancel' for safe exit

**Example (from planner.py):**
```python
print("\nAvailable sport types:")
for idx, sport in enumerate(sport_list, 1):
    count = len(workouts_by_sport[sport])
    print(f"{idx}. {sport.capitalize()} ({count} workouts)")

choice = input("> ").strip().lower()
if choice == "q":
    return 0
```

## Workout API & Data Models

### Garmin Workout Structure
Workouts returned from `client.get_available_workouts()` contain:
```python
{
    "workoutId": int,
    "workoutName": str,
    "sportType": {
        "sportTypeKey": str,  # e.g., "running", "cycling"
        "sportTypeId": int,
        "displayOrder": int
    },
    # ... other fields
}
```

**Important:** Extract sport type via `workout.get("sportType", {}).get("sportTypeKey", "unknown").lower()` — do NOT use `sportTypeName` (it doesn't exist).

### Deleting Workouts
- Use `garth.request("DELETE", "connectapi", f"/workout-service/workout/{workout_id}")`
- Returns 204 No Content on success
- 403 Forbidden = User can't delete (API permission issue)

## Git Workflow

Follow **Conventional Commits** format:


### Structure
- **Type:** `feat`, `fix`, `docs`, `refactor`, `test`, `chore`

1. Run `git status` to see changes
2. Review via `git diff` (visual inspection recommended)
3. `git add -A` to stage all
4. `git commit -m "..."` with meaningful message
5. `git push` to origin/main

## Logging & Error Handling

### Logging Standards
- Use `LOGGER = logging.getLogger(__name__)` at module top
- Levels: `DEBUG` (verbose), `INFO` (normal), `WARNING` (recoverable), `ERROR` (failed operation)
- Include context: `LOGGER.error("Failed to delete workout with ID %s: %s", workout_id, exc)`

### Error Messages
- Be descriptive: "API returned 403 Forbidden — user may not have permission to delete"
- Don't expose sensitive data (passwords, tokens)
- Suggest next steps when possible

1. Add argument to `_build_parser()` in cli.py
2. Add handler in `main()` with conditional
3. Implement feature method in planner.py
4. Update README (both languages)
5. Test with `--dry-run` first

### Adding an API Integration
1. Implement in `garmin_client.py` (pure API calls)
2. Orchestrate in `planner.py` (business logic)
3. Expose via `cli.py` argument
4. Log all API calls with context

### Handling Interactive Input
1. Display options with numbers (1, 2, 3)
2. `choice = input("> ").strip().lower()`
3. Validate range
4. Confirm destructive actions
5. Handle 'q'/'cancel' for exit

## Known Limitations & Workarounds

- **Workout Scheduling:** Not available via API — users must schedule manually in Garmin Connect web/app
- **403 Forbidden on Delete:** May occur if workout is locked or user lacks permission — document in error message
- **Token Expiry:** Tokens valid ~1 year; users can manually delete `~/.garminconnect` to re-authenticate
